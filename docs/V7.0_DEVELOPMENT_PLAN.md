## 🏗️ 一、技术架构决策

### 1.1 整体架构

- **前端**：Next.js 14 + React 18 + TypeScript + Tailwind CSS + Framer Motion + Zustand（完全复用现有）
- **新增能力**：Three.js（可视化）、Web Audio API（音频特效）、MediaRecorder（录制）
- **桌面层**：Electron（并行开发，与 Web 层完全隔离，通过 IPC 通信）
- **后端**：FastAPI + Python 3.11+（仅在必须时扩展，前端优先）

### 1.2 代码隔离策略

```
src/
├── components/
│   ├── (现有组件保持不变)
│   ├── visualization/          # 新增：可视化相关组件（独立目录）
│   ├── desktop-lyrics/         # 新增：桌面歌词相关组件
│   └── features-v7/            # 新增：V7.0 所有新功能组件
├── hooks/
│   ├── (现有 hooks 保持不变)
│   └── useVisualization.ts     # 新增：可视化相关 hooks
├── store/
│   ├── (现有 store 保持不变)
│   ├── visualizationStore.ts   # 新增：可视化状态
│   └── desktopLyricsStore.ts   # 新增：桌面歌词状态
└── pages/
    └── visualization.tsx       # 新增：可视化独立路由（可选）
```

## 🎨 二、界面交互设计

### 2.1 视图切换机制

- **扩展现有 uiStore**：在 `currentView` 枚举中新增 `'visualization'`
- **切换入口**：在现有播放器界面添加「可视化」按钮（不修改播放器 UI 核心布局）
- **过渡动画**：复用现有 Framer Motion 过渡效果
- **完全隔离**：可视化组件独立渲染，不侵入现有三个界面的代码

### 2.2 弹窗/面板策略

所有新增专业功能（音频处理、智能分析等）继续使用现有弹窗模式：
- 复用 `GlassPanel` 组件风格
- 通过状态控制显示/隐藏
- 不修改现有界面布局

## 🛡️ 四、风险控制与降级策略

### 4.1 功能降级策略

| 功能 | 风险 | 降级方案 | 用户提示 |
|-----|------|---------|---------|
| WebGL 可视化 | 兼容性、性能 | 切换到简单 CSS 可视化 | 友好提示，提供切换选项 |
| WASM 音频处理 | 加载失败 | 降级到 Web Audio API | 提示用户部分特效不可用 |
| 桌面歌词 | Electron 失败 | 回退到 Web 弹窗歌词 | 提示桌面歌词不可用 |
| 大库操作 | 性能瓶颈 | 启用虚拟滚动、后端索引 | 提示正在处理，请稍候 |

### 4.2 异常处理流程

1. **文件异常**：尝试自动修复 → 失败则提示用户 → 用户选择跳过/重试/删除
2. **功能失败**：记录日志 → 自动降级 → 友好提示用户
3. **性能瓶颈**：监控资源使用率 → 超过阈值自动暂停非核心功能 → 提示用户

### 4.3 数据迁移策略

- **自动迁移**：首次启动检测数据格式，自动升级
- **备份机制**：迁移前自动备份，支持回滚
- **向后兼容**：新数据结构设计确保完全兼容旧版本

## 🔄 六、开发流程与审核机制

### 6.1 严格审核策略

1. **分支管理**：
   - 主分支 `main` 严格保护
   - 所有开发在 `feature/v7.0` 分支进行
   - 每个子功能创建独立分支 `feature/v7.0-xxx`

2. **代码审查**：
   - 每个功能完成后提交 PR
   - 用户审查确认：**特别确认现有三个界面未被修改**
   - 确认后合并到 `feature/v7.0`
   - 阶段完成后合并到 `main`

3. **进度汇报**：
   - 每个子功能完成后即时汇报
   - 每个阶段完成后统一汇报
   - 提供 demo 或截图供确认

### 6.2 后续迭代

- V7.0 完成后，先让用户使用一段时间
- 根据实际使用反馈决定下一步开发方向
- 不预设后续迭代计划，反馈驱动
